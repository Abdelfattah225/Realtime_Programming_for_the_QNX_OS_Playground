# QNX Message Passing - Pulses

## Overview

A **pulse** is a small, non-blocking message used for lightweight notifications in QNX.

### Key Characteristics

| Property | Description |
|----------|-------------|
| **Non-blocking** | The sender does **not** wait for the pulse to be received or replied to |
| **Unidirectional** | "Send and forget" — no reply from server back to client |
| **Small payload** | Fixed-size data fields only |
| **Faster & cheaper** | More efficient than sending a small message with an empty reply |

> **Important:** Pulses are ideal for simple wake-up or change notifications. They are **not** a substitute for sending large amounts of data. Sending 1024 pulses to carry 4KB of data is not efficient.

---

## Pulse Data Fields

A pulse carries **two fixed-size data fields**:

```
┌─────────────────────────────────────┐
│  code  (8-bit)  │  value  (64-bit)  │
└─────────────────────────────────────┘
       ▲                  ▲
   Pulse type        Associated data
```

| Field | Size | Description |
|-------|------|-------------|
| `code` | 8-bit | Identifies the **type** of pulse (similar to message types) |
| `value` | 64-bit | Associated data — can be treated as a `pointer`, `long` (64-bit), or `int` (32-bit) |

**Total usable payload: 71 bits** (64-bit value + 7-bit code)

---

## Pulse Codes and Ranges

The 8-bit `code` field has a range of **-128 to +127**.

| Range | Usage |
|-------|-------|
| `-128` to `-1` | **Reserved** — QNX kernel system use (e.g., client process died notifications) |
| `0` to `+127` | ✅ **Available** for user/application use (7 usable bits) |

### Defined Constants for Safe Range

```c
// Use these defines for safe pulse code ranges
_PULSE_CODE_MINAVAIL   // minimum available pulse code (user range)
_PULSE_CODE_MAXAVAIL   // maximum available pulse code (user range)
```

> **Rule:** Use only **positive** pulse codes (`_PULSE_CODE_MINAVAIL` to `_PULSE_CODE_MAXAVAIL`) for user-defined pulses. Negative codes are reserved for kernel use.

## Sending Pulses

### `MsgSendPulse(coid, priority, code, value)`

Sends a pulse to a server channel.

```c
MsgSendPulse(coid, priority, code, value);
```

| Parameter | Description |
|-----------|-------------|
| `coid` | Connection ID — identifies which channel and process receives the pulse |
| `priority` | Delivery priority for the server thread handling the pulse |
| `code` | 8-bit pulse type (use positive values only) |
| `value` | 32/64-bit data payload |

### Priority Behavior

| Priority Value | Effect |
|----------------|--------|
| `-1` | Uses the **calling thread's current priority** |
| `1` to `253` | Uses the specified **explicit priority** |

> **Note:** Unlike messages (where the server inherits the client thread's priority directly), pulses store the priority with the pulse data because the client thread may be gone or changed priority by the time the pulse is delivered.

### `MsgSendPulsePtr(coid, priority, code, ptr)`

A variant of `MsgSendPulse()` that treats the value field as a **`void*` pointer** instead of an integer — makes pointer passing more convenient.

```c
MsgSendPulsePtr(coid, priority, code, my_ptr);
```

### Privilege Requirements

Sending pulses to **other processes** requires appropriate privileges:

| Scenario | Requirement |
|----------|-------------|
| Same user ID | Allowed — no special privilege needed |
| Different user ID | Requires **root privilege** or appropriate **QNX ability** |

> **Security Note:** This privilege requirement prevents **denial-of-service attacks** where a malicious process spews pulses at another process. Messages do not have this requirement because servers can naturally flow-control message processing by delaying replies.

## Receiving Pulses

Pulses are received using the **same `MsgReceive()` call** as regular messages — they share a common receive point.

### Distinguishing Pulses from Messages

The **return value** of `MsgReceive()` tells you what was received:

| Return Value | Meaning |
|-------------|---------|
| `> 0` | Received a **message** — return value is the `rcvid` (identifies client thread/process) |
| `= 0` | Received a **pulse** — you cannot reply to this |
| `= -1` | Error occurred |

```c
rcvid = MsgReceive(chid, &msg, sizeof(msg), &info);

if (rcvid > 0) {
    // It's a regular message — process and MsgReply()
} else if (rcvid == 0) {
    // It's a pulse — do NOT call MsgReply()
} else {
    // Error
}
```

> ⚠️ **Warning:** Do **not** pass `rcvid = 0` to `MsgReply()`. It will return an error.

### Pulse Data Structure: `struct _pulse`

When a pulse is received, data is written to your receive buffer as a `struct _pulse` — a fixed OS-defined size of **24 bytes**.

| Field | Description |
|-------|-------------|
| `code` | The pulse code sent via `MsgSendPulse()` |
| `value` | The associated data (union of `long`, `int`, or `void*`) |
| `scoid` | **Server Connection ID** — identifies which client process sent the pulse |

```c
// Accessing pulse fields after receiving
if (rcvid == 0) {
    int   pulse_code  = msg.pulse.code;
    int   pulse_value = msg.pulse.value.sival_int;   // as int
    long  pulse_long  = msg.pulse.value.sival_ptr;   // as pointer
    int   pulse_scoid = msg.pulse.scoid;             // sender's connection ID
}
```

> **Note:** The `scoid` can be used with `ConnectClientInfo()` to get more information about the client process that sent the pulse.

### ⚠️ Buffer Size Requirement

Your receive buffer **must** be at least **24 bytes** to hold a `struct _pulse`.

If the buffer is too small:
- The kernel dequeues the pulse from its internal queue
- Fails to fit the data into the buffer
- Returns `-1` with `errno = EFAULT`
- **The pulse data is permanently lost**

### Recommended Pattern: Union Receive Buffer

Declare your receive buffer as a **union** to guarantee it is always large enough for a pulse:

```c
// messages.h or server.c

typedef union {
    struct _pulse   pulse;        // guarantees minimum size of 24 bytes
    my_msg_a_t      msg_a;        // your custom message type A
    my_msg_b_t      msg_b;        // your custom message type B
    // ... other message types
} myMessage_t;
```

> **Why this works:** The C language guarantees that a `union` is at least as large as its largest member. Including `struct _pulse` ensures the buffer is always large enough.

```c
myMessage_t msg;

// Safe: sizeof(msg) >= sizeof(struct _pulse) always
rcvid = MsgReceive(chid, &msg, sizeof(msg), NULL);
```

## Pulse Queuing and Memory Management

### Default Behavior

By default, pulses are **queued until the system runs out of memory**.

**Risks of unlimited queuing:**
- Dropped pulses may not be detectable
- Draining system memory can affect other processes
- A misbehaving server can starve the system

### Pre-Allocated Pulse Pool

To control memory usage, replace `ChannelCreate()` with `ChannelCreatePulsePool()`:

```c
struct nto_channel_config config;

// Set the number of pulses to pre-allocate
config.num_pulses = 32;   // pre-allocate queue entries for 32 pulses

chid = ChannelCreatePulsePool(flags, &config);
```

| Behavior | Description |
|----------|-------------|
| Pre-allocated entries | Guarantees space for the specified number of pulses |
| Pool exceeded | **Process is killed** by default (no other way to detect overflow) |
| Identical pulses | Automatically **conglomerated** into one queue entry with a count |

### Pulse Conglomeration (Optimization)

The kernel optimizes pulse queuing by **combining identical pulses** into a single queue entry with a count:

```
Identical pulses (same code + value + priority + origin):
[pulse A] [pulse A] [pulse A]  →  [pulse A × 3]   (1 queue entry)

Different pulses:
[pulse A] [pulse B] [pulse C]  →  3 separate queue entries
```

**Identical** means: same `code` + same `value` + same `priority` + same `origin`

> **Key Point:** The pre-allocated pool size is a **guaranteed minimum**. Identical repeated pulses can effectively queue more than the allocated count due to conglomeration.

## Pulse vs Message Comparison

| Feature | Pulse | Message |
|---------|-------|---------|
| Blocking | Non-blocking (sender) | Blocking (sender waits for reply) |
| Direction | Unidirectional | Bidirectional |
| Data size | Fixed 71 bits (code + value) | Variable size |
| Reply | ❌ No reply | ✅ MsgReply() required |
| Priority | Stored with pulse data | Inherited from client thread |
| Speed | Faster & cheaper | Slower (copy + reply overhead) |
| Cross-process privilege | Required (different UID) | Not required |
| Flow control | ❌ None | ✅ Server controls via reply timing |
| Receive function | `MsgReceive()` (rcvid == 0) | `MsgReceive()` (rcvid > 0) |

## Server-Side Pulse Handling

### Complete Server Loop with Pulse and Message Handling

```c
#include <sys/neutrino.h>
#include "messages.h"

typedef union {
    struct _pulse   pulse;
    myMessage_t     msg;
} recv_buf_t;

void server_loop(int chid) {
    recv_buf_t  buf;
    my_reply_t  reply;
    int         rcvid;

    while (1) {
        rcvid = MsgReceive(chid, &buf, sizeof(buf), NULL);

        if (rcvid == -1) {
            // Handle error
            break;
        }

        if (rcvid == 0) {
            // ─── PULSE received ───────────────────────────────
            // Do NOT call MsgReply() here
            switch (buf.pulse.code) {

                case MY_PULSE_CODE_A:
                    // Handle pulse type A
                    handle_pulse_a(buf.pulse.value.sival_int);
                    break;

                case MY_PULSE_CODE_B:
                    // Handle pulse type B
                    handle_pulse_b(buf.pulse.value.sival_ptr);
                    break;

                case _PULSE_CODE_DISCONNECT:
                    // System pulse: client disconnected
                    handle_disconnect(buf.pulse.scoid);
                    break;

                default:
                    // Unknown pulse — ignore
                    break;
            }

        } else {
            // ─── MESSAGE received ─────────────────────────────
            // Must call MsgReply() or MsgError()
            switch (buf.msg.base.type) {

                case MY_MSG_TYPE_A:
                    process_message_a(&buf.msg);
                    MsgReply(rcvid, EOK, &reply, sizeof(reply));
                    break;

                default:
                    MsgError(rcvid, ENOSYS);
                    break;
            }
        }
    }
}
```

### Key Differences: Pulse Handling vs Message Handling

| Aspect | Message | Pulse |
|--------|---------|-------|
| Reply | **Required** — `MsgReply()` or `MsgError()` | **Forbidden** — no reply possible |
| Trigger | `rcvid > 0` | `rcvid == 0` |
| Data access | `buf.msg.type` | `buf.pulse.code` and `buf.pulse.value` |
| Client info | Available via `info` struct | Available via `buf.pulse.scoid` + `ConnectClientInfo()` |
